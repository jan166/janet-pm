01 · Why

비전 모델이 먹을 프레임의 "진짜 전처리 경계"

Clova EyeD는 신사옥 1784의 출입 FaceSign 카메라, eKYC의 모바일 카메라, 일본 PayPay의 인증 단말기 등 다양한 센서 위에서 같은 모델이 돌아야 합니다. 센서 · 렌즈 · ISP 단계를 넘어 들어오는 이미지가 제각각이면 모델이 "편향되지 않는 입력"을 받지 못합니다. "전처리는 어디서 시작하는가"라는 질문을 소프트웨어 밖으로 확장해, 카메라 세팅 · 편광 필터 · 센서 보정까지 포함한 전체 전처리 스택을 정리한 리서치 프로젝트.

GOAL 01

전처리 경계의 재정의

"카메라 세팅도 전처리인가?"에 답을 내리고, 하드웨어-소프트웨어 경계에 있는 처리 단계들의 목록 확정.

GOAL 02

1784 FaceSign 이슈 해결

출퇴근 시간대 역광 조건에서의 얼굴 검출 실패를 하드웨어/소프트웨어 양쪽에서 접근.

GOAL 03

미래 센서 대비

화소가 점점 작아지는 추세 — 현재 전처리로 커버 안 되는 미래 요구를 예견해 이미징 아키텍처 설계의 입력으로 활용.

02 · The Real Preprocessing Pipeline

전처리는 소프트웨어에서 시작되지 않는다

비전 모델 입력까지의 전처리를 다섯 단계로 풀어내면, 3개 단계가 이미 센서/ISP 안에서 끝나 있습니다. 모델 팀이 건드릴 수 있는 영역은 마지막 2단계뿐 — 그래서 초기 3단계 설계가 잘못되면 소프트웨어로 회복 불가.

STAGE 01

센서 · 렌즈

CCD/CMOS로 빛→전기신호 변환. 렌즈 구면 형상 · 렌즈 셰이딩이 이미 입력 편향 생성.

STAGE 02

카메라 세팅

노출 · 화이트밸런스 · 편광 필터 유무. 이 시점에 데이터를 걸러 찍는 의미에서 이미 전처리.

STAGE 03

ISP (센서 내부)

셰이딩 보정 · 불량화소 · 암전류 제거 등. raw format 유지하며 센서/이미지 프로세서가 처리.

STAGE 04

SW 전처리

리사이즈 · 크롭 · 색공간 변환 · 정규화. 모델 팀이 주로 관리하는 영역.

STAGE 05

모델 입력

Face Detection · Keypoint · Recognition 등. 여기에 들어오는 프레임의 품질이 여기서 좌우.

03 · ISP Corrections

센서 안에서 이미 일어나는 5가지 보정

모델 팀에게 "raw 이미지"로 들어오기 전에 이미 센서 · ISP가 수행하는 보정들. 이 단계 각각에서 설계 실수가 있으면 소프트웨어로 뒤집기 어렵습니다.

보정 종류 왜 필요한가 (원인) 처리 방식 / 메모
Shading Compensation
셰이딩 보정		 렌즈 구면 형상 때문에 중심부와 가장자리의 도달 광량이 다름. 균일한 밝기 피사체조차 가장자리가 어두워 보이는 왜곡 발생. 서로 다른 셰이딩을 보이는 복수 영상에서 동일 위치 값들을 비교해 보정 함수를 생성, 이후 영상에 일괄 적용.
Dead Pixel Correction
불량 화소 보정		 생산 시 발견 못한 초기 불량 + 노후 · 온도 · 습도 · 먼지 · 전원 이상으로 발생하는 Hot / Dead / Stuck Pixel. 주변 화소 값을 기반으로 결함 픽셀을 추정/대체. 센서 레벨 상시 보정.
Dark Current Elimination
암전류 제거		 빛이 없어도 흐르는 전류 노이즈. 저조도 환경에서 특히 문제. 무광 상태 캘리브레이션 프레임으로 노이즈 베이스라인 측정 후 차감.
Pixel Linearization
화소 선형화		 센서 응답이 비선형 — 밝기에 따라 감도가 다름. 룩업 테이블 기반 선형화로 후단 프로세싱의 예측 가능성 확보.
Green Imbalance Filtering
녹색 불균형 필터링		 베이어 패턴의 Gr · Gb 간 응답 차이로 체커 패턴 아티팩트 발생. 필터링 단계에서 Gr/Gb 정합 → 고주파 아티팩트 제거.
04 · 1784 Case Study

네이버 신사옥의 FaceSign 카메라를 다시 짠다

1784는 네이버 신사옥 · 테크 컨버전스 빌딩. 출입/근태를 FaceSign으로 처리합니다. 이 환경에서 가장 자주 터진 이슈는 출퇴근 시간대의 역광 조건에서 얼굴 검출 실패. "데이터만 더 넣으면 된다"는 소프트웨어적 해결을 넘어, 특화 디바이스(RGB+Thermal · 편광 필름)로 하드웨어 차원에서도 접근했습니다.

HARDWARE 01

RGB + Thermal 듀얼 카메라

1784 FaceSign 디바이스는 RGB 카메라에 Thermal 카메라를 병행 탑재. Thermal로 얼굴 영역을 1차 확보해 RGB 검출 실패 시의 안전장치를 구성.

→ 모델 이중 스택: Thermal ROI → RGB Fine-grained

HARDWARE 02

편광 필름 실험

일반 카메라로 조절 불가능한 빛 반사 · 역광 조건을 하드웨어 수준에서 필터링. 편광의 방향/정도를 독립 조절해 이미지가 PC로 전송되기 전에 오브젝트 특징이 살아나도록 함.

→ 1784 카메라에 편광 필름 부착 실험 · 역광 검출 개선 확인

SOFTWARE 03

역광 데이터 개별 어노테이션

하드웨어 접근으로 다 해결되지 않는 변수는 원거리 역광 데이터 개별 어노테이션으로 트레인셋 확장. 모델이 역광 조건을 직접 학습.

→ 데이터 파이프라인과 연계 (Vision Data Ops 참조)

05 · My Contribution

정효주 기여 영역

모델 팀이 익숙한 "데이터 + 학습 루틴" 프레임을 넘어, 센서/ISP/특화 디바이스 층까지 포함한 전처리 스택을 PM이 이해하고 정리한 R&D 리서치. 기획이 이 영역을 알고 있어야 하드웨어 팀과 모델 팀 사이의 의사결정이 "누구 책임이냐"가 아닌 "어디서 풀 것이냐"로 정렬됩니다.

R&D Research · Hardware-aware Planning

전처리 경계를 하드웨어까지 확장한 리서치

  • "전처리는 어디서 시작하는가"에 답을 내린 파이프라인 정의 — 센서 · 카메라 세팅 · ISP · SW 전처리 · 모델 입력의 5단계 파이프라인으로 스택을 해체.
  • ISP 5개 보정 정리 — Shading · Dead Pixel · Dark Current · Pixel Linearization · Green Imbalance Filtering. 각각 원인과 해결법을 표로 문서화해 모델 팀과 하드웨어 팀의 공통 언어 제공.
  • 1784 FaceSign 이슈 대응 — 출퇴근 역광 검출 실패 이슈를 하드웨어/소프트웨어 이중 접근으로 재정의. RGB+Thermal 듀얼 · 편광 필름 실험을 데이터 어노테이션 확장과 병행.
  • 편광 카메라 실험 설계 지원 — 일반 카메라로는 조절 불가능한 반사/역광 이슈를 하드웨어 레벨에서 필터링하는 경로를 탐색. 실험 근거와 기대 효과 문서화.
  • Vision Data Ops와 연결 — 하드웨어가 잡지 못하는 변수는 데이터 구축 룰로 흡수하도록, 역광 조건 별도 어노테이션을 Vision Data Ops의 구축 가이드에 반영.

요약

Camera & Image Processing Research는 "비전 모델이 먹을 프레임은 어디서부터 전처리되는가"를 해체한 리서치 프로젝트였습니다. 센서 · 카메라 세팅 · ISP · SW 전처리 · 모델 입력의 5단계 파이프라인을 정리하고, 센서 내부에서 이미 일어나는 5가지 보정(셰이딩 · 불량화소 · 암전류 · 선형화 · 녹색불균형)을 PM이 이해 가능한 형태로 문서화했습니다. 네이버 신사옥 1784 FaceSign 카메라의 역광 검출 이슈에는 RGB+Thermal 듀얼 · 편광 필름 실험 · 역광 데이터 개별 어노테이션의 이중 접근으로 대응. 이 리서치는 이후 Clova EyeD가 하드웨어 맥락이 다른 복수 고객사(1784, eKYC 모바일, 일본 결제 단말 등)에 배포될 때 "어디까지 하드웨어에서 흡수할 것인가"의 의사결정 기반이 되었습니다.
