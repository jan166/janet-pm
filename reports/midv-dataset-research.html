01 · Why

왜 공개 데이터셋을 먼저 뜯어 봐야 했나

Clova EyeD는 일본 PayPay · LinePay · 네이버 파이낸셜 등 금융/eKYC 환경에 배포되는 SDK입니다. "신분증을 정확히 인식했다"는 주장을 하려면 공정한 벤치마크가 있어야 하는데, 신분증 자체가 민감정보라 자체 데이터로만은 내부 확신을 넘어서는 신뢰도를 만들기 어려웠습니다. MIDV 시리즈는 학계에서 Wikimedia의 sample/specimen 문서와 퍼블릭 데이터만으로 이 문제를 우회한 몇 안 되는 공개 데이터셋이었고, 이 설계 논리를 그대로 Clova EyeD의 평가셋 설계 원칙으로 이식할 수 있다고 판단했습니다.

CONSTRAINT 01

민감정보 · 공개 불가

실제 신분증은 개인정보 포함으로 외부 공유 불가. 모델 성능의 외부 검증 경로가 사실상 막혀 있음.

CONSTRAINT 02

학계 벤치마크 부재

Face / OCR / 문서 영역에 비해 IDCard 특화 공개 셋이 극히 제한적. Face Detection · Text Line · Document Data Extraction 세 개 태스크를 한번에 다룰 수 있는 셋이 없음.

CONSTRAINT 03

실환경 조건 결여

MIDV-500조차 왜곡·조명 변수는 고려되지 않음. 모바일 eKYC 시나리오의 핵심 난이도를 담기 부족.

02 · Dataset Breakdown

MIDV-500 → MIDV-2019, 무엇이 바뀌었나

두 데이터셋을 나란히 놓고 보면 학계가 "실환경 조건을 하나씩 추가해가며" 벤치마크를 성숙시킨 흐름이 보입니다. 이 확장의 축이 곧 Clova EyeD의 평가셋이 다뤄야 할 변수의 목록이 됩니다.

항목 MIDV-500 (2018) MIDV-2019
비디오 클립 500개 500 + 200개 신규 = 700+ 고화질 클립
ID 종류 50종 (ID·여권·운전면허 등) 동일 50종 유지 (17 ID · 14 여권 · 13 면허 · 6 기타)
촬영 환경 테이블 · 키보드 · 손 · 일부 가림 · 잡동사니 배경 + Projective Distortion · 어두운 조명 조건 추가
촬영 디바이스 아이폰 · 갤럭시 (실모바일 환경) 고화질 비디오 클립 (해상도 업)
데이터 출처 Wikimedia sample/specimen 문서 · 퍼블릭 데이터만 동일 원칙 승계 (개인정보 회피)
한계 왜곡 · 조명 변수 미반영 일부 해소 · 블러/포커스/반사는 여전히 약함

원 논문: arxiv.org/pdf/1807.05786 · FTP: smartengines.com/midv-500/extra/midv-2019/

03 · Five Evaluation Axes

Clova EyeD 평가셋이 다뤄야 할 5개 변수

MIDV-500의 한계("왜곡 · 조명이 없다")와 MIDV-2019의 보완분을 종합해, eKYC SDK 평가셋이 반드시 교차 검증해야 하는 축을 5개로 정리했습니다. 이 축은 곧 Clova EyeD 평가셋의 샘플링 룰 · 알체라 외주 데이터 구축 가이드의 근거가 됩니다.

AXIS 01

저화질 (Low-resolution)

저가 단말/저조도 상황 카메라 노이즈. MIDV-500의 폰 촬영분과 유사 · 이미 커버됨.

MIDV 커버
AXIS 02

조명 (Lighting)

어두운 실내 · 역광 · 플래시. MIDV-2019에서 어두운 조명 추가되었으나 역광/스팟라이트는 부족.

부분 커버자체 확장
AXIS 03

블러 (Motion / Focus Blur)

손떨림 · 셔터 타이밍. MIDV 계열에서 체계적 커버 안 됨 → 자체 데이터에 명시 포함 필요.

자체 확장
AXIS 04

포커스 (Out-of-focus)

피사체 근접 실패 · AF 지연. 블러와 구분해 관리 — 원인이 다르므로 실패 모드도 다름.

자체 확장
AXIS 05

반사 (Reflection)

홀로그램 · 코팅 반사 · 스크린 반사. eKYC 현장의 가장 빈번한 실패 원인 중 하나 — 반드시 확장.

자체 확장
OUTCOME

평가셋 샘플링 룰

5축 × 신분증 종류를 교차 매트릭스로 구성, 모든 셀에 최소 샘플 수를 강제해 평가 편향 제거.

04 · Multi-task Utility

하나의 셋, 세 개의 태스크

MIDV 시리즈의 가장 큰 설계 이점은 단일 데이터셋으로 3개 태스크를 동시에 평가할 수 있다는 점입니다. Clova EyeD가 다루는 태스크 구성과 1:1로 정렬되어, 그대로 내부 평가셋 설계 템플릿으로 이식했습니다.

TASK A

Face Detection

신분증 내 얼굴 영역 검출. 포커스/조명 조건이 까다로울수록 실패율이 드러남.

→ Clova EyeD Face SDK 평가 루틴과 정렬

TASK B

Text Line Recognition

신분증 내 문자 라인 인식. 왜곡·반사가 섞일 때 OCR 성능 저하 지점을 드러냄.

→ Clova OCR / eKYC 통합 파이프라인 평가와 정렬

TASK C

Document Data Extraction

라벨/필드 추출. MRZ · 성명 · 생년월일 · 발행국 등 구조화 필드 추출 정확도 검증.

→ eKYC 최종 결과물 정합성 평가와 정렬

05 · My Contribution

정효주 기여 영역

데이터 조사 → 평가 설계 → 외주 데이터 사양으로 이어지는 업스트림 연결을 담당. "공개 데이터셋을 쓸 수 있는가"가 아니라 "공개 데이터셋의 설계 논리를 우리 데이터 사양에 어떻게 이식할 것인가"로 문제를 재정의했습니다.

Data Strategy · Product Planning

공개 데이터셋의 설계 논리를 내부 평가셋 원칙으로 이식

  • MIDV-500/2019 논문 · FTP 아카이브 정독 — 700+ 클립 · 50종 ID 구성의 샘플링 설계, Wikimedia specimen 기반 데이터 수집 원칙, projective distortion 추가 의도 분석.
  • MIDV 한계 → Clova EyeD 자체 데이터 요구사항 역산 — 왜곡 · 조명 · 블러 · 포커스 · 반사 5개 평가축을 도출해 eKYC SDK 평가셋의 최소 커버리지 조건으로 확정.
  • 3-태스크 평가 템플릿 정렬 — Face Detection · Text Line · Document Extraction 3개 태스크가 단일 셋으로 교차 평가되도록, Clova EyeD 내부 평가 파이프라인 구조와 1:1 정렬.
  • 민감정보 회피 설계 가이드 — Wikimedia specimen 원칙을 사내 데이터 구축 시 적용할 수 있는 형태로 정리, 외주(알체라) 계약서의 데이터 수집 조항 근거로 활용.

요약

MIDV Dataset Research는 신분증이라는 공개 불가능한 도메인에서 eKYC 모델을 공정하게 평가하기 위한 리서치 프로젝트였습니다. 학계의 MIDV-500 / MIDV-2019를 정독해 700+ 클립 · 50종 ID · Wikimedia specimen 기반 설계 논리를 해체하고, 그 한계(왜곡 · 조명 · 블러 · 포커스 · 반사)를 5개 평가축으로 역산해 Clova EyeD의 내부 평가셋 설계 원칙과 외주 데이터 구축 가이드의 근거로 이식했습니다. 단일 데이터셋으로 Face Detection · Text Line · Document Extraction 3개 태스크를 동시에 평가하는 템플릿은 이후 Clova EyeD 평가 파이프라인 구조와 그대로 정렬되었습니다.
