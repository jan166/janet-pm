01 · Why

왜 벤치마킹부터 했나

당시 사내 ML 인프라는 NSML · CVA 등 개별 도구가 따로 돌고 있었고, 학습/서빙/데이터 라벨링/파이프라인을 한 제품 아래에서 묶는 통합 플랫폼은 아직 없었습니다. "우리도 플랫폼을 만들자"라는 논의가 있었지만, 무엇을 먼저 만들지 정하려면 업계 표준 UX를 한 번 돌려 본 사람이 필요했습니다. Vertex AI는 AutoML · Custom Training · Feature Store · Monitoring · Pipelines · Metadata까지 MLOps 전 스펙트럼을 한 콘솔에서 제공하는 가장 최신 사례였기에, 이를 실제로 돌려보고 피처 단위로 해부하는 벤치마킹 리서치를 진행했습니다.

PURPOSE 01

Flova 피처 설계의 근거

사내 ML 플랫폼이 필수로 가져야 할 피처 목록을 외부 표준과 대조해 확정.

PURPOSE 02

UX 레퍼런스

"설명서 없이도 순서가 보이는 대시보드" 같은 UX 원칙을 직접 체험해 사내 플랫폼 정보 구조 설계에 반영.

PURPOSE 03

갭 분석

Vertex의 각 기능을 사내 자산(NSML · CVA · PAF · NCP 등)과 매핑해 갭을 시각화, 우선순위 제안.

02 · End-to-End PoC

신분증 분류 모델을 Vertex에서 직접 돌려봤다

페이퍼로만 읽어서는 얻을 수 없는 것들이 있기에, "주민등록증/운전면허증을 구분하는 이미지 분류 모델"을 Vertex AI 위에서 처음부터 끝까지 직접 구축했습니다. Dataset 생성 → AutoML 학습 → Endpoint 배포 → 단건/배치 테스트까지 전 구간을 실측해, 실제 운영 시 예상되는 병목과 UX 특성을 기록했습니다.

STEP 01

Dataset 생성

이미지/테이블/텍스트/동영상 유형 선택. 어노테이션 방식 변경 가능.

STEP 02

이미지 가져오기

PC 업로드 또는 Cloud Storage 참조. 학습용 Cloud Storage로 일원화.

STEP 03

라벨링 작업

라벨 정의 · 가이드 업로드 · 작업자 수 설정. 권장 샘플 100장+.

STEP 04

AutoML 학습

Train/Valid/Test 분할 + 학습 자원 지정. 20장 기준 18분 39초.

STEP 05

Evaluation

Test Set 기반 Precision/Recall by threshold. 소규모 셋에서는 정보량 제한.

STEP 06

Endpoint 배포

엔드포인트 이름 + 컴퓨팅 지정. 몇 클릭으로 서빙. GCP 로그 연계 자동.

STEP 07

단건/배치 테스트

UI에서 이미지 업로드 또는 API 호출. 배치는 GCS 경로 지정.

STEP 08

모니터링

Google Cloud 로깅과 연계. 데이터 편향/편차 alert 자동.

CONSOLE NOTE 01

지원 리전 제약

대시보드에서 리전을 '서울'로 설정하면 아직 사용 불가능한 기능이 다수. 글로벌 리전 우선 설계를 전제로 기능이 롤아웃되는 구조 — 사내 플랫폼은 역순이 되어야 한다는 판단 근거.

CONSOLE NOTE 02

라벨링 데이터 업로드 제약

이미 라벨링된 결과물을 그대로 업로드할 수는 없음. 단순 이미지만 올릴 수 있고, 라벨링은 Vertex 내부에서만 수행. 외부 라벨링 도구와의 연동이 막힌다는 한계.

CONSOLE NOTE 03

학습 시간 최소치

AutoML 최소 20시간 / Auto Edge ML 최소 1시간의 강제 하한. 작은 데이터셋 PoC라도 비용 · 시간 구조를 예측 가능하게 만드는 설계지만, 단발성 실험에는 무거운 편.

CONSOLE NOTE 04

일괄 예측의 번거로움

테스트 데이터를 JSON으로 GCS에 업로드 → 예측 → JSON으로 반출. 한 장 테스트도 꽤 느리고, 테스트 결과를 콘솔에서 바로 보여주는 UI는 부재. 일회성 검수용 UX 보강 여지가 있음.

04 · UX Insights

PM이 실제로 배운 UX 원칙 5가지

피처 매핑은 숫자놀이지만, 콘솔을 직접 쓰는 몇 시간이 피처 리스트로는 절대 보이지 않는 설계 원칙을 드러냈습니다. Flova/사내 ML 플랫폼 UX 회의에서 반복 인용된 원칙 5개를 정리.

INSIGHT 01

Dashboard에서 "순서"가 보여야 한다

Dataset → Training → Inference의 흐름을 설명서 없이도 이해할 수 있도록 Dashboard가 스스로 안내. 초심자에게 학습 곡선을 낮추는 핵심 UX.

INSIGHT 02

Storage 경로의 일원화

이미지 · 데이터셋 · 모델 패키지가 전부 Cloud Storage 경로로 일원화. 모델과 데이터는 동일 리전 강제. "저장 위치의 단일성"이 운영 복잡도를 크게 낮춤.

INSIGHT 03

Dataset : Model = 1 : 1

Raw data가 다양해도 Dataset이라는 중간 단계에서 필요한 것만 골라 합친 뒤, Dataset과 Model을 1:1로 묶어 학습. 관리 주체의 책임 경계가 명확.

INSIGHT 04

Background job의 상태 복원력

업로드 · 로딩 · 학습 같은 백그라운드 작업이 화면을 벗어났다 돌아와도 상태가 정확. 장시간 작업에서 사용자 신뢰를 유지하는 기반.

INSIGHT 05

배포는 "몇 번의 클릭"이어야 한다

엔드포인트 이름 + 노드 타입만 지정하면 배포. GCP 기존 서비스(로그 · 모니터링)와 자동 엮임. 배포 단계를 "선택의 문제"로 축소시키는 설계.

COUNTER

사내 제약 — 학습 속도

Vertex는 학습 데이터를 항상 GCS에서 로딩. 사내는 학습 속도 이슈로 Object Storage(Nubes, Verda)를 쓰지 않음. "경로 일원화" 원칙은 좋지만, 네이버는 스토리지 구조부터 다른 제약이 있음을 확인.

05 · My Contribution

정효주 기여 영역

"ML 플랫폼을 만들자"라는 논의에 실제 사용 경험과 피처 매핑 근거를 제공하는 프로덕트 리서처 역할. 슬라이드 한 장의 벤치마크가 아니라, 콘솔을 직접 돌려본 사람만 쓸 수 있는 리포트를 남겼습니다.

요약

Vertex AI Benchmarking은 사내 ML 플랫폼(Flova 후보) 기획의 기초자료를 만들기 위한 리서치 프로젝트였습니다. Google I/O'21에 공개된 Vertex AI를 신분증 분류 모델 PoC로 end-to-end 구축(20장 학습 · 18분 39초 · 22/21 정확도)해 실측 경험을 확보했고, 이를 근거로 Vertex의 16+ 기능을 사내 자산과 1:1로 매핑해 사내 미보유 8건의 갭을 시각화했습니다. Dashboard의 순서성 · Storage 경로 일원화 · Dataset:Model 1:1 · 원-클릭 배포 등 5개 UX 원칙은 이후 사내 ML 플랫폼 UI 설계 논의의 공통 레퍼런스가 되었습니다.
