01 · Product Vision

왜 만드는가

업무 중 정보를 찾으려면 Wiki · Slack · 커넥트 · 로컬 폴더 · 인트라넷을 각자의 창에서 각자의 방식으로 검색해야 합니다. 그 과정에서 "맥락이 끊긴다"는 문제를 한 단계에서 해결하는 것이 Launcher의 목표입니다. 여기에 HCX를 얹어 단순 검색을 넘어 질문 → 답변 → 실행까지의 흐름을 데스크탑 키보드 단축키 하나로 완결시킵니다.

VISION 01

업무의 연속성 유지

현재 작업창을 벗어나지 않고도 가장 빠른 경로로 원하는 정보에 도달. ⌘+E 단축키 하나로 시작되는 검색 UX로 컨텍스트 스위칭 비용을 제거.

VISION 02

RAG × LLM 시나리오 강화

RAG Platform과 연동해 사내 데이터를 기반으로 HCX가 답변. 단순 "검색된 문서 목록"이 아니라 질문에 대한 답 그 자체를 반환.

VISION 03

SaaS 진입점의 일원화

SSO 기반으로 타 사내 SaaS로의 진입점을 하나로. Launcher가 곧 "팀네이버 업무 허브"가 되도록 설계.

02 · Core Flow

3 스텝 인터랙션

데스크탑의 어느 화면에 있든 ⌘+E로 호출 → 검색 또는 HCX 질문 → 답변/링크로 바로 연결. "앱을 켠다"는 단계 자체를 제거한 것이 가장 큰 UX 전환점입니다.

STEP 01

호출

어느 앱에 있든 단축키로 Launcher 호출. 현재 작업창을 가리지 않는 플로팅 UI.

⌘ + E
STEP 02

검색 또는 질문

키워드를 입력하면 통합 검색. HCX 버튼을 누르면 같은 입력창이 질문창으로 전환.

Keyword · Prompt
STEP 03

결과 · 실행

검색 결과는 원본 링크로 바로 점프. HCX 답변은 출처와 함께 인라인 노출, 후속 질문 가능.

Open · Follow-up
03 · Product Features

핵심 피처 3블록

통합 검색 · HCX 어시스턴트 · 생산성 유틸리티. 이 세 축이 Launcher를 구성하고, RAG Platform 연동으로 검색의 의미 수준을 한 단계 끌어올립니다.

S

통합 검색 (v0.5)

흩어진 사내 데이터를 한 입력창으로

  • 로컬 파일 검색 (파일명 · 폴더)
  • Slack 메시지 · 파일 · 채널
  • 사내 Wiki 페이지 (Confluence)
  • 커넥트 업무 일정 · 게시물
  • 사원 정보 조직도 · 연락처
H

HCX 어시스턴트 (v0.6)

클릭 한 번으로 검색을 LLM 질문으로

  • 입력창 내 HCX 버튼 토글로 모드 전환
  • 자연어 질문 → 답변 인라인 표시
  • 후속 질문으로 대화 이어가기
  • 기본 프롬프트 · 시스템 가이드 내장
  • 향후 RAG Platform 결합으로 사내 지식 기반 답변 고도화 예정
U

생산성 유틸리티

검색 외 업무의 간극을 메우는 도구들

  • 뉴스피드 (예정) — 구독 기반 업무 피드
  • SSO 진입점 — 사내 SaaS 단일 허브
  • Quick launch · 빈번 사용 문서 핀
  • 향후 업무 유틸 지속 확장
04 · Release Roadmap

버전별 릴리즈 계획

초기 통합 검색 기반을 먼저 완성한 뒤, HCX · RAG Platform · 유틸을 단계별로 쌓아 올려 최종적으로 팀네이버 전사 공개를 목표로 하고 있습니다.

v0.5 릴리즈 완료
통합 검색 MVP — 로컬 파일 · Slack · Wiki · 커넥트 · 사원정보 5개 소스 통합 검색. ⌘+E 단축키 기반 플로팅 UI 완성.
v0.6 릴리즈 완료
HCX 기본 기능 탑재 — HCX 버튼을 통한 LLM 질문 모드 제공. 검색창을 질문창으로 전환하는 UI 인터랙션 구축.
v0.7+ 예정
유틸리티 확장 — 생산성 유틸 추가 (퀵 노트, 즐겨찾기 등), 검색 결과 UX 고도화.
v1.0 예정
RAG Platform 연동 · 뉴스피드 — 사내 데이터 기반 RAG 답변 제공, 업무 구독형 뉴스피드 추가. HCX 응답이 단순 LLM 답변을 넘어 사내 지식 기반 인용 답변으로 진화.
Team NAVER GA 예정 (연내 목표)
팀네이버 전사 공개 — 사용성 · 안정성 강화, HCX 연동 시나리오 확장. SSO를 통한 사내 SaaS 진입점 통합까지 완성.

06 · My Contribution

정효주 기여 영역

AI Solution 조직의 기획 리소스로 Launcher 프로덕트 방향성 수립과 RAG Platform 연동 설계의 접점을 담당. 엔터프라이즈 서치 마켓 리서치에서 출발해, 피처 구조 · 사용자 여정 · 경쟁사 포지셔닝까지 프로덕트의 뼈대를 그렸습니다.

Product Planner · AI Solution

엔터프라이즈 서치 마켓 리서치 · 포지셔닝 설계

  • 글로벌 경쟁사 9곳 심층 분석 — Mindbreeze · Lucidworks · Sinequa · Elastic · Coveo · Lucy · OpenText · Yext · EPAM의 비즈니스 전개 방식, Killer Feature, 브랜딩, 고객사, 경쟁 우위 포인트를 매트릭스로 비교.
  • 검색의 4단계 모델 정의 — "검색 전 / 검색 / 검색 중 / 결과물" 각 단계별 단계 특성 · Painpoint · 경쟁사 매핑 · "우리가 잘할 수 있는 포인트"를 도출해 차별화 축을 데이터 연결 · Hallucination 없는 RAG · 결과 커스터마이징으로 정의.
  • User Journey 2종 설계 — "필요한 자료를 찾아서 공유/업무 진행" 여정과 "최소 시간으로 최대 생산성 결과물 제작" 여정을 각각 단계별 핵심행동 · 유저 혼란 수준(1~5) · Needs/Painpoint · 개선 포인트 · Feature로 풀어냄.
  • 피처 후보군 도출 — search bookmark · one click search · 0-depth search · 1hr 인수인계 솔루션 · 원본 그대로 보여주는 인용 솔루션 · export customization 등 Launcher의 초기 피처 풀을 User Journey에서 역산.
Cross-project Bridge

RAG Platform × Launcher 연동 설계 브릿지

  • RAG Platform을 Launcher의 HCX 시나리오 강화 재료로 정의 — Launcher의 특장점 중 "기업내부 정보를 이용한 RAG Platform과의 연동을 통한 LLM 사용 시나리오 강화"를 핵심 포지션으로 프로덕트 문서에 확정.
  • RAG 도입 시나리오 재사용 — RAG Platform 기획에서 진행한 Nomura · 사내 ESS Orchestration 실험 결과를 Launcher의 HCX 답변 품질 기준으로 역-참조해 "단순 검색 결과 리스트"가 아닌 "인용 기반 답변"이 왜 필요한지의 근거 마련.
  • HCX 답변 포맷 가이드 — ESS Orchestration 개선 피드백에서 확인된 "넘버링 · 볼드 · 목차 구조"가 유용성에 미치는 영향을 Launcher의 HCX 답변 UX에도 동일하게 반영하도록 전달.

요약

Launcher (shook)는 팀네이버 직원이 데스크탑 어디에서든 ⌘+E 한 번으로 사내 정보(로컬 파일 · Slack · Wiki · 커넥트 · 사원정보)를 찾고 HCX로 바로 질문까지 이어갈 수 있는 설치형 업무 어시스턴트입니다. v0.5 통합 검색 MVP, v0.6 HCX 기본 기능이 릴리즈 완료되었고, 이어서 RAG Platform 연동 · 뉴스피드 · 팀네이버 전사 공개를 향해 로드맵을 확장 중. AI Solution(기획 2.5MM · FE 1.2MM)과 뉴로서치(BE 1.0MM)가 결합된 4.7MM의 슬림한 구성으로, 정효주는 AI Solution의 기획 리소스로 엔터프라이즈 서치 마켓 리서치 · User Journey 설계 · 피처 도출 · RAG Platform 연동 포지셔닝을 담당했습니다.
